# 功能实现验证及代码校验

在完成功能修改或实现后，对修改后的代码进行检查和校验。确保功能已全部实现，且实现后的代码逻辑清晰、结构正确、无冗余。

## 何时执行

- 用户完成功能修改或新功能实现后，要求做验证或校验
- 用户提到「功能实现验证」「代码校验」「检查实现是否完整」「逻辑是否清晰」「是否有冗余」

## 校验流程

### 1. 功能完整性检查

- 对照需求或任务说明，逐项确认是否均已实现
- 确认新增/修改的入口（路由、事件、接口调用）已正确接入
- 确认边界与异常情况有处理（空值、错误态、加载态等）
- 若涉及 UI，确认交互与展示符合预期（必填、禁用、成功/失败反馈等）

### 2. 逻辑与结构校验

- **逻辑清晰**：分支、循环、异步流程易于理解，无明显矛盾或重复判断
- **结构正确**：职责单一，模块/组件/函数划分合理，依赖关系清晰
- **命名一致**：变量、函数、文件命名与项目现有风格一致且表意明确

### 3. 冗余与质量

- 删除未使用的变量、函数、导入、样式或 dead code
- 合并重复逻辑为可复用函数或公共模块
- 不保留仅用于调试的 console、注释掉的整块代码或临时 hack

### 4. 输出校验结果

按下面结构给出结论，便于用户逐项确认，其中检查项使用✔和❌来展示结果：

```markdown
## 功能实现验证及代码校验结果

### 功能完整性
- [ ] 需求/任务项是否全部实现
- [ ] 入口与集成是否正确
- [ ] 边界与异常是否处理
- [ ] UI/交互是否符合预期

### 逻辑与结构
- [ ] 逻辑是否清晰、无矛盾
- [ ] 模块/函数职责与结构是否合理
- [ ] 命名是否一致、表意明确

### 冗余与质量
- [ ] 是否存在未使用代码
- [ ] 是否存在可合并的重复逻辑
- [ ] 是否已清理调试代码与临时写法

### 问题与建议
（列出发现的问题及具体修改建议，若无则写「无」）
```

## 执行原则

- 只针对本次修改或实现所涉及的文件与范围进行校验，不扩大无关范围
- 发现问题时指出具体位置（文件、函数/组件名、行号或代码片段）并给出可操作的修改建议
- 若项目有 `.cursor/rules` 或编码规范，校验时一并遵守
