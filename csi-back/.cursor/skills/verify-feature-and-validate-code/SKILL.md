# 功能实现验证及代码校验

在完成功能修改或实现后，按下列步骤对修改后的代码进行检查和校验。

## 执行时机

- 用户完成某次功能开发、修改或实现之后
- 用户明确要求「验证功能」「代码校验」「检查实现」「确认无冗余」时

## 校验流程

按顺序执行以下检查，并给出结论与修改建议。

### 1. 功能完整性

- 对照需求或任务描述，逐项确认是否均已实现
- 确认边界条件、异常路径、可选参数等是否被覆盖
- 如有遗漏，列出未实现项及建议实现方式

### 2. 逻辑正确性

- 梳理核心逻辑（分支、循环、递归、状态变更）是否与预期一致
- 检查条件判断是否完整、无矛盾，无逻辑漏洞
- 确认数据流与依赖关系正确（入参、出参、副作用）

### 3. 结构合理性

- 模块/函数职责是否单一、边界清晰
- 命名是否与职责一致，是否与项目现有风格一致
- 层级与依赖是否合理，是否存在循环依赖或不必要的耦合

### 4. 冗余与质量

- 是否存在未使用的变量、导入、函数或死代码
- 是否存在重复逻辑，可否抽取复用
- 是否存在过长函数、过深嵌套，是否影响可读性

## 输出格式

校验完成后，用以下结构汇总（可节选适用部分），其中检查项使用✔和❌来展示结果：

```markdown
## 功能实现验证及代码校验结果

### 功能完整性
- [ ] 已全部实现 / 存在遗漏
- （若有遗漏）未实现项：…

### 逻辑与结构
- 逻辑：清晰 / 需改进（简要说明）
- 结构：合理 / 需改进（简要说明）

### 冗余与质量
- 冗余代码：无 / 有（位置与建议）
- 其他建议：…

### 结论
- 通过 / 需修改后再校验
```

## 注意事项

- 仅针对本次修改或实现所涉及的代码与功能范围进行校验，避免无限扩大范围
- 结论与建议要具体到文件、函数或行级，便于修改
- 若项目有既定规范（如 .cursor/rules、lint 配置），校验时应一并考虑
