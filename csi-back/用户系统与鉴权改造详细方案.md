# 用户系统与鉴权改造详细方案（后端 + 前端）

## 1. 目标与范围

### 1.1 改造目标

- 建立系统级用户体系（登录、登出、用户状态管理）。
- 建立可扩展的权限体系（角色组 + 权限码），支持页面级与按钮级控制。
- 支持数据按用户或组织维度隔离展示。
- 前后端统一鉴权协议，避免仅前端隐藏导致越权。

### 1.2 本次建议范围

- 后端：认证（Authentication）+ 授权（Authorization）+ 基础 RBAC + 数据范围控制。
- 前端：登录态接入 + 路由守卫 + 菜单可见性控制 + 按钮级权限控制。
- 不包含：复杂工作流审批、多租户计费、SSO（可后续扩展）。

---

## 2. 当前项目现状摘要（基于现有代码）

### 2.1 后端现状（`csi-back`）

- 使用 FastAPI，统一响应包装（`ResponseMiddleware`）。
- 已定义 `UnauthorizedException`、`ForbiddenException`，但未形成完整鉴权链路。
- `app/models/action/accounts.py` 的 `AccountModel` 是任务/站点账号，不是系统用户。
- 持久层具备 MongoDB（Beanie）+ Redis 基础能力。
- 目前无登录接口、无 token 校验依赖、无权限校验依赖。

### 2.2 前端现状（`csi-front`）

- Vue3 + Vue Router + Axios。
- 已有“系统配置/用户权限管理”页面骨架（含 mock 数据）。
- 路由未配置鉴权守卫，菜单未按权限过滤。
- 请求层未注入 `Authorization`，未统一处理登录失效。
- 尚无统一 auth store（未使用 Pinia/Vuex）。

## 3. 架构设计建议

## 3.1 认证方式建议

采用“分阶段”认证策略：

阶段一（优先落地）：

- 仅启用 `JWT Access Token`（短期有效，如 30 分钟），随请求头传递。
- 先不启用自动刷新，过期后前端统一引导重新登录。

阶段二（安全增强）：

- 再引入 `Refresh Token`（如 7~30 天）与轮换机制。
- Redis 存储 refresh token 白名单/版本号，支持主动登出与会话失效控制。

说明：

- 该策略可显著降低首期联调复杂度，先形成登录鉴权闭环，再增强会话治理。

## 3.2 授权模型建议（RBAC + 权限码）

模型建议：

- 用户（User）
- 角色组（Role）
- 权限（Permission）
- 用户-角色关系（UserRole）
- 角色-权限关系（RolePermission）

权限码命名建议：

- 页面级：`page:system:permissions`
- 功能级：`agent:config:view`
- 操作级（按钮）：`agent:config:create`、`agent:config:edit`、`agent:config:delete`

核心原则：

- 前端负责“显示/隐藏/禁用”。
- 后端负责“最终放行/拒绝”。

## 3.3 数据范围模型建议

为了“不同用户看到不同数据”，建议引入数据范围字段（按业务场景选择）：

- 单用户私有数据：`owner_id`
- 组织共享数据：`org_id`
- 可见性控制：`visibility`（`private` / `org` / `public`）

查询规则示例：

- 普通用户：仅可查本人 `owner_id == current_user.id`
- 组织管理员：可查本组织 `org_id == current_user.org_id`
- 超级管理员：可查全部

## 4. 后端详细改造方案

## 4.1 建议新增目录结构

建议在 `app` 下新增：

- `app/models/auth/`：用户、角色、权限模型
- `app/schemas/auth/`：登录/用户信息/权限返回结构
- `app/service/auth.py`：认证与权限核心逻辑
- `app/core/security.py`：密码哈希、JWT 签发与校验
- `app/api/v1/endpoints/auth.py`：登录、登出、刷新、当前用户
- `app/dependencies/auth.py`：`get_current_user`、`require_permissions`

## 4.2 配置项新增（`app/core/config.py` + `.env`）

建议新增配置：

- `AUTH_SECRET_KEY`
- `AUTH_ALGORITHM`（如 `HS256`）
- `ACCESS_TOKEN_EXPIRE_MINUTES`
- `REFRESH_TOKEN_EXPIRE_DAYS`
- `AUTH_ISSUER`（可选）
- `AUTH_AUDIENCE`（可选）
- `AUTH_ENABLE_REGISTER`（可选）

## 4.3 数据模型建议（字段级）

模型边界固定约束（必须遵守）：

- 权限与鉴权域模型统一归属 `auth` 领域：`User`、`Role`、`Permission`、`UserRole`、`RolePermission`。
- 系统用户统一使用 `user` 语义，不使用 `account` 命名。
- 禁止复用 `app/models/action/accounts.py`（任务/站点账号）承载系统登录与权限能力。

### 4.3.1 User

- `id`（主键）
- `username`（唯一）
- `email`（唯一，可选）
- `password_hash`
- `display_name`
- `enabled`（布尔）
- `is_superuser`（布尔，可选）
- `org_id`（可选）
- `last_login_at` / `last_login_ip`
- `created_at` / `updated_at`

### 4.3.2 Role

- `id`
- `role_key`（唯一，如 `admin`、`analyst`）
- `role_name`
- `enabled`
- `remark`

### 4.3.3 Permission

- `id`
- `perm_key`（唯一，如 `agent:config:edit`）
- `perm_name`
- `module`（可选）
- `enabled`

### 4.3.4 关系表

- `user_role(user_id, role_id)`
- `role_permission(role_id, permission_id)`

### 4.3.5 Token 会话（可选但推荐）

Redis Key 设计建议：

- `auth:refresh:{user_id}:{device_id}` -> refresh_token_hash / version
- `auth:blacklist:access:{jti}` -> 过期时间

## 4.4 接口设计建议

### 4.4.1 认证接口

- `POST /auth/login`
  - 入参：`username` + `password`
  - 返回：`access_token`、`refresh_token`、`user`、`permissions`
- `POST /auth/refresh`
  - 入参：`refresh_token`
  - 返回：新 `access_token`（可选轮换 refresh）
- `POST /auth/logout`
  - 入参：当前 token 或 refresh_token
  - 处理：移除 refresh 白名单，必要时黑名单 access jti
- `GET /auth/me`
  - 返回：当前用户信息 + 角色 + 权限码列表

分阶段实现说明：

- 阶段一：优先实现 `POST /auth/login`、`POST /auth/logout`、`GET /auth/me`。
- 阶段二：再启用 `POST /auth/refresh` 与自动续期机制。

### 4.4.2 用户与权限管理接口（系统配置页）

- `GET /system/users`
- `POST /system/users`
- `PUT /system/users/{id}`
- `PATCH /system/users/{id}/status`
- `GET /system/roles`
- `POST /system/roles`
- `PUT /system/roles/{id}`
- `GET /system/permissions`
- `PUT /system/roles/{id}/permissions`
- `PUT /system/users/{id}/roles`

## 4.5 FastAPI 依赖与拦截建议

### 4.5.1 认证依赖

- `get_current_user`：
  - 从 `Authorization: Bearer <token>` 解析并校验。
  - 校验用户状态（禁用用户禁止访问）。
  - 使用 FastAPI `HTTPBearer` 作为安全依赖注入（`Security(HTTPBearer())`），以便 OpenAPI 正确声明 Bearer 认证方案，Swagger `/docs` 可通过右上角 `Authorize` 一次性填写 token，并自动在后续请求中携带 `Authorization` 请求头。

### 4.5.2 权限依赖

- `require_permissions(["agent:config:edit"])`
  - 用户无权限时抛 `ForbiddenException`。

### 4.5.3 路由接入方式

- 在需要保护的接口上增加：
  - `current_user = Depends(get_current_user)`
  - 或 `Depends(require_permissions([...]))`

## 4.6 数据范围控制接入方式

服务层统一封装数据过滤逻辑：

- `build_data_scope_filter(current_user, action)`
- 在列表、详情、修改、删除时统一调用，避免漏校验。

示例策略：

- `view_all` 权限：可查看全部
- `view_org` 权限：可查看组织内
- 默认：仅本人

## 4.7 响应规范统一建议（已按新编码规范调整）

当前系统沿用“HTTP 200 + code 字段”的风格，统一约定如下：

- 所有接口（含鉴权失败）统一返回 HTTP 200。
- 通过响应体 `code` 区分成功/失败类型，响应结构固定为 `code`、`message`、`data`。
- 前端仅基于 `code` 做登录失效、无权限、提示与跳转处理。
- `code=0` 表示成功；非 0 统一按 6 位编码规则输出。

状态码统一编码规则（后续版本保持不变）：

- 最高位为定义位：`1` 表示系统定义内部状态码，`2` 表示符合 HTTP 标准状态码。
- 第 `2-4` 位是状态码分类值（3 位大类）。
- 第 `5-6` 位是子分类（当前默认 `00`）。
- `0` 保持不变。
- 若原始值是 HTTP 标准状态码（如 `404`），编码为 `2 + 404 + 00`，即 `240400`。
- 若原始值是非标准正数（如 `22`），编码为 `1 + 022 + 00`，即 `102200`。
- 若原始值是非标准负数（如 `-2`），先取绝对值并补齐三位，再将三位分类值最高位强制为 `1`，即 `102 -> 110200`。

权限与鉴权相关固定业务码（按新规则）：

- `240100`：token 缺失/无效（原 `40101` 的语义迁移到 `401` 类别统一编码）
- `240300`：权限不足
- `240400`：资源不存在或目标对象不可见

实现要求：

- 后端所有错误码统一经编码函数归一化后返回，不允许绕过统一转换直接下发裸码。
- `HTTPException`、自定义异常、业务返回错误统一进入同一编码规则。
- 前端仅按新 6 位编码处理登录失效、无权限与异常提示。

## 4.8 安全建议

- 密码必须强哈希存储，不可逆。
- 登录接口增加限流（用户名 + IP 维度）。
- 关键操作记录审计日志（操作者、目标、前后值、时间、IP）。
- JWT 不存敏感明文信息（如密码、手机号）。

## 5. 前端详细改造建议（结合 `csi-front` 现状）

## 5.1 状态管理方案

考虑当前项目未接 Pinia，可先做轻量方案：

- 新增 `src/stores/auth.js`（响应式对象）
  - `token`
  - `user`
  - `permissions`
  - `roles`
  - 方法：`setAuth`、`clearAuth`、`hasPermission`

后续如果状态复杂再迁移 Pinia。

## 5.2 请求层改造（`src/utils/request.js`）

### 5.2.1 请求拦截

- 自动注入 `Authorization`：
  - `config.headers.Authorization = Bearer ${token}`

### 5.2.2 响应拦截

- 识别登录失效（业务码 `401xx`）：
  - 清理本地登录态
  - 跳转登录页
  - 保留原目标路由用于登录后回跳

- 识别权限不足（业务码 `403xx`）：
  - 跳转 403 页或展示统一无权限提示

### 5.2.3 刷新 token（可选）

- 在响应拦截中做一次自动刷新重试。
- 注意并发场景只发起一次刷新请求（队列或锁）。

## 5.3 路由改造（`src/router/index.js`）

### 5.3.1 为路由增加元信息

- `meta.requiresAuth: true/false`
- `meta.permissions: []`

示例：

- `/system/permissions` 需要 `page:system:permissions`
- `/agent/engine-config` 需要 `agent:config:view`

### 5.3.2 增加全局守卫 `beforeEach`

校验流程：

1. 判断是否需要登录。
2. 未登录 -> 跳登录页。
3. 已登录但缺权限 -> 跳 403 页（或首页）。
4. 允许访问 -> next。

## 5.4 菜单与页面显隐（`Header.vue` 等）

现有 `Header.vue` 菜单是静态渲染，建议改为配置驱动：

- 抽取导航配置数组（含 `requiredPermissions`）。
- 渲染前根据 `hasPermission` 过滤。

建议效果：

- 无权限不显示菜单项。
- 有入口权限但无子功能权限时，子菜单部分显示。

## 5.5 按钮级权限控制

建议统一一套方式，避免 scattered if：

方案 A（快速）：

- 各按钮使用 `v-if="hasPermission('xxx')"`

方案 B（推荐）：

- 封装 `Permission` 组件或 `v-permission` 指令：
  - `<Permission code="agent:config:edit"><el-button>编辑</el-button></Permission>`

适配策略：

- 删除类高风险操作：无权限直接隐藏。
- 可读但不可执行操作：可显示禁用态并给提示。

## 5.6 登录页与用户信息展示

新增页面：

- `/login`
- 可选 `/403`

Header 右上角账号区建议改为：

- 展示用户名/昵称
- 下拉菜单：个人信息、退出登录

## 5.7 系统权限管理页联调（`UserPermissionManagement.vue`）

当前为效果展示 mock 数据，字段可按后端接口重新设计，建议分阶段：

1. 先确定页面展示 DTO（前端展示字段）与后端返回 DTO（接口字段）映射
2. 替换用户列表为 `GET /system/users`
3. 替换权限组列表为 `GET /system/roles`
4. 增加用户分配角色、角色绑定权限的弹窗与保存接口
5. 处理分页、搜索、启用禁用状态

字段建议：

- 页面层可保留 `uuid/create_at/login_date` 等展示字段；
- 接口层建议统一使用 `id/created_at/last_login_at`，由前端做一次映射转换，避免后端为展示命名妥协。

## 6. 前后端联调协议建议

## 6.1 `/auth/me` 返回建议

返回结构建议包含：

- `user`: 基本资料
- `roles`: `["admin"]`
- `permissions`: `["page:system:permissions", "agent:config:edit"]`

## 6.2 前端权限判断约定

- 只依据 `permissions` 做判断。
- `roles` 仅用于展示，不参与前端权限逻辑（减少前后端规则分叉）。

## 6.3 错误处理约定

- `240100`：未登录或登录失效，前端清空登录态并跳登录。
- `240300`：已登录但无权限，前端进入 403 页面或提示。
- `24xxxx`：其余标准状态码映射错误，按业务提示处理。
- `10xxxx/11xxxx`：系统内部自定义码，按模块提示处理。
- HTTP 状态码保持 200，不作为前端鉴权判断依据。

## 7. 迁移与实施步骤（推荐执行顺序）

## 阶段 0：准备

- 明确权限码清单与命名规范。
- 明确哪些页面/按钮需要控制。

## 阶段 1：后端基础能力

- 完成用户、角色、权限模型与基础数据初始化。
- 完成登录、登出、`/auth/me`，先不启用 refresh。
- 完成 `get_current_user` 与 `require_permissions`。

## 阶段 2：前端登录态接入

- 完成 auth store、请求头注入、`401xx/403xx` 处理。
- 增加登录页与路由守卫。

## 阶段 3：页面级权限

- 给路由配置 `meta.permissions`。
- Header 与导航按权限过滤。

## 阶段 4：按钮级权限

- 优先改造高风险操作按钮（删除、发布、配置修改）。
- 补齐后端同权限校验。

## 阶段 5：数据范围控制

- 先选择 1~2 个核心模块试点接入 `owner_id/org_id` 策略。
- 验证查询正确性与性能后，再逐步扩展到其他模块。
- 对历史数据进行回填与兼容（可先默认 `owner_id=admin`）。

## 阶段 6：审计与安全增强

- 关键操作审计日志。
- 登录失败限流与告警。

## 8. 验收清单

## 8.1 功能验收

- 未登录访问受保护页面被正确拦截。
- 登录后可见菜单与权限一致。
- 无按钮权限时无法显示或无法执行。
- 直接调用受限接口返回 403。
- 数据列表仅返回允许范围内的数据。

## 8.2 安全验收

- 密码库中无明文。
- token 过期可被识别并处理。
- 登出后 token 不可继续使用（阶段一至少保证 access 不再放行；阶段二保证 refresh 不可用）。
- 关键权限变更有审计记录。

## 8.3 性能验收

- 权限校验对主链路延迟可控。
- 菜单与按钮权限判断不引入明显渲染卡顿。

---

## 9. 风险与规避

- 风险：仅前端做权限隐藏导致越权  
  规避：所有写接口必须后端二次校验权限。

- 风险：历史数据无 owner 字段导致范围查询困难  
  规避：制定回填脚本与过渡策略。

- 风险：权限码定义混乱，长期维护成本高  
  规避：统一命名规范，按模块分组管理。

- 风险：token 刷新并发导致重复请求  
  规避：前端刷新锁 + 请求排队重放。

## 12. 实施建议结论

建议采用“分层分阶段”方案：

1. 先跑通认证与页面级权限，首期使用 Access Token 并按 `code` 协议处理。
2. 再扩展按钮级权限和数据范围控制，采用先试点后全量的推进方式。
3. 最后补 refresh 会话治理、审计与限流，提升安全与可运维性。

这样可以在不大改现有架构的前提下，稳步接入你需要的用户系统和细粒度权限能力。

## 13. 远期功能升级：在线用户表/会话表

在首期我们采用“无状态 access token + 后端实时校验用户 enabled/is_deleted”的方式，通常不需要维护在线用户表。
当你完成当前所有目标后，若出现更强的会话治理、审计与强制下线诉求，可以再引入“在线用户表/会话表”（建议以 Redis 存储为主，或加一张 MongoDB 会话集合做持久化）。

### 13.1 在线用户表/会话表要支持的功能

- **登出立即失效**：为每次登录创建会话记录，并在 `logout` 时将该会话标记为无效（从而让未过期的 access token 也立即拒绝）。
- **Refresh token 轮换与吊销**：当后续启用 refresh token 时，可以对 refresh 会话进行版本号/白名单治理，实现“使用后轮换、老 token 失效、主动登出全失效”。
- **并发会话限制**：为单用户/单设备设置最大会话数，超出时拒绝或踢出旧会话。
- **强制踢下线**：管理员可根据用户或会话维度执行“踢下线”，并触发前端回跳登录。
- **会话审计与排查**：记录登录 IP、设备指纹、创建/失效时间、操作来源，便于安全审计与故障排查。
- **在线状态统计**：支持后台展示“当前在线人数/在线用户列表（按时间窗口或会话有效期计算）”，用于运维分析与告警。

### 13.2 什么时候维护在线用户表/会话表会变得有价值

- **你要求 `logout` 必须“立刻”生效**（而不仅依赖 access token 的 `exp` 过期）。
- **你计划启用 refresh token**，并需要实现“轮换、版本号校验、主动吊销”来降低会话劫持风险。
- **你需要并发控制/强制下线**，例如同一用户最多登录 N 次、管理员可一键踢掉风险会话。
- **你需要更完善的安全审计**（合规/风控要求会话维度的留痕与可追溯）。
- **你需要运维可观测性**：例如按会话维度定位异常请求来源、统计在线活跃度、生成告警信号。

> 建议的取舍原则：只有当出现上述“会话治理/审计/强制失效”的明确业务需求时，再引入会话表；否则继续保持首期的无状态策略，会更简单也更易维护。
