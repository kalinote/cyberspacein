# CSI 权限系统重构方案

## 1. 文档目的

本文档定义 CSI 权限系统的目标设计、实施范围、改造清单、数据清理目标、测试要求与验收标准，用于指导当前开发周期内的权限系统重构。

本文档描述系统行为和实施计划，不约束具体类名、函数名、数据库驱动或代码组织方式。除明确列入“后续目标”的内容外，其余要求在同一个开发周期内完成。

## 2. 背景与结论

CSI 整套系统部署和使用于内网，不向公网开放。但内网边界不作为安全边界，所有业务接口仍必须在应用层完成身份认证和权限校验。

当前系统已经具备用户、权限组、权限码和前端权限控制的基础结构，但存在以下核心问题：

- 后端仅有少量接口接入认证和授权，前端权限可以被直接调用 API 绕过；
- 前后端权限码定义不统一，`view/use` 语义和兼容权限并存；
- 权限字典、后端硬编码权限和前端权限之间缺少唯一事实源；
- 权限组可以保存任意权限，超级权限和授权范围缺少保护；
- 当前 Token 无法支持服务端会话、立即失效、多端管理和权限版本刷新；
- 单个操作的 403 会触发全局跳转，部分 Tab 授权无法正常使用。

本次重构采用以下总体方向：

1. 全局管理能力使用 RBAC；
2. 业务资源最终使用“RBAC + 资源所有权或工作区成员关系”；
3. 本期先完成全局 RBAC、认证会话和后端接口保护，资源隔离留作后续实现；
4. 使用 YAML 维护后端有效权限的唯一事实源；
5. 使用标准 JWT 表示登录凭证，并通过服务端会话状态实现立即失效和多端管理；
6. 页面权限统一使用 `visible/access`，数据权限统一使用 `read/create/update/delete/execute`；
7. 不兼容旧权限码，开发环境清理权限相关数据后重新初始化。

### 2.1 现状盘点（2026-07-18）

本轮实施前已对当前仓库完成只读盘点，结论如下：

- 后端 `app/api/v1/endpoints` 中共有 153 个 HTTP/WebSocket 路由装饰器，其中只有 14 个路由函数显式接入认证或权限依赖，且主要集中在 Auth 和 System；其余业务接口尚未形成应用层认证边界；
- Action SDK 的初始化、结果上报和心跳 3 个端点当前均为匿名访问，SDK 也没有携带组件身份；
- 当前访问凭证是自定义的两段式 `payload.signature`，不是标准三段 JWT；凭证只包含用户、签发和到期信息；
- 当前退出登录不撤销服务端状态，系统没有登录会话记录、具体会话终止、多端管理、凭据版本或授权版本；
- 当前用户权限仅为所有启用组中权限字符串的并集，没有与权限字典启用状态、后端标准权限注册表或可委派范围求交集；
- 当前权限管理只保护少量接口，普通组可以保存任意字符串，系统账号、系统组和 `*` 缺少完整的业务保护；
- 用户资料、密码、用户组、状态、临时账号和到期时间仍可通过同一个通用更新接口修改；
- 用户名、权限组标识和权限码当前只有普通索引，没有数据库唯一约束；
- 前端仍以 `view/use` 权限为主，存在 page 权限兼容 operation 权限的逻辑；权限配置为空或类型错误时部分工具默认放行；
- 当前前端收到任意操作级 403 都会跳转全局 403，尚未实现授权版本、窗口聚焦刷新和单个 Tab 按权限加载。

以上数据作为本次重构的实施基线。后续接口矩阵检查以框架实际注册路由为准，不仅依赖源文件正则统计。

### 2.2 已确认的实现决策

经实施前确认，本期采用以下具体方案：

1. 用户登录会话持久化到 MongoDB，登录失败计数、账号临时锁定和来源速率限制使用 Redis；
2. 普通用户访问凭证使用标准三段式 HS256 JWT，服务端仍以 MongoDB 会话和实时权限计算为准；
3. Action Component 使用独立的短期服务 JWT，凭证绑定行动 ID、节点实例 ID、用途和允许的 SDK 接口，并结合节点状态实现提前撤销；为避免完整 JWT 进入 Crawlab 任务参数，调度时只传递 5 分钟有效、一次性消费的引导码，引导码在 Redis 中仅保存哈希，SDK 交换成功后才获得组件 JWT；
4. 新增显式 `ENVIRONMENT=development/test/production` 配置，不再使用 `DEBUG` 推断环境安全边界；
5. 权限数据清理能力只作为仓库内独立脚本交付，不挂载 API、不随服务启动、不集成前端；
6. 清理脚本仅允许在 `development` 环境运行，执行前打印目标数据库、集合和 Redis namespace，并要求操作者显式确认；本轮实现和测试脚本，但不自动执行当前 `csi_db` 的清理。

- 不实现显式拒绝；

- 不改变敏感值当前允许返回前端的总体策略；
- 不实现密码复杂度、历史密码和首次登录强制改密。

## 4. 系统访问边界

### 4.1 公开端点

本期仅允许以下端点匿名访问：

- 登录接口；
- Swagger UI；
- OpenAPI 描述接口。

健康检查如果存在，默认也需要认证；后续如需供基础设施匿名探活，应单独评审并限制返回内容。

### 4.2 默认保护规则

- 所有业务路由默认要求有效登录会话；
- 新增路由如果没有明确声明为公开，自动继承认证要求；
- 页面和按钮权限只用于前端交互，不能替代后端授权；
- 后端对每一次读取、写入和执行操作进行权限判断；
- 内网、反向代理、Crawlab 或 API 网关均不能替代应用层鉴权。

### 4.3 Action SDK 边界

Action SDK 的初始化、结果上报和心跳由 `base_components` 通过 `base_components_sdk` 调用，不属于普通用户会话，也不得匿名开放。

本期推荐采用独立的组件身份认证：

- 后端在调度节点实例时签发短期、最小权限的组件凭证；
- 凭证绑定行动、节点实例和允许调用的 SDK 接口；
- 不允许组件凭证访问普通用户 API；
- 不允许普通用户 JWT 调用仅供组件上报的接口；
- 凭证必须可撤销、可过期，并能够防止跨节点复用；
- 本期确定采用独立短期服务 JWT。JWT 必须使用与普通用户凭证不同的用途约束，绑定行动和节点实例，并由 SDK 在初始化、结果上报和心跳请求中携带。后端除校验签名和过期时间外，还必须校验节点存在、行动绑定一致、节点处于允许上报的状态，并拒绝跨节点复用。调度参数和日志不得输出完整凭证。
- 由于当前 Crawlab 单次运行接口的 [`SpiderRunOptions`](https://pkg.go.dev/github.com/crawlab-team/crawlab/core/interfaces#SpiderRunOptions) 只提供 `cmd/param/mode/node_ids/priority` 等运行选项，不提供任务级私密环境变量注入，调度参数采用一次性引导码而不是完整 JWT。引导码随机生成、短时有效、绑定行动和节点、Redis 仅保存其哈希，并在首次交换时原子删除；多个关联组件分别获得独立引导码。引导码不能直接访问初始化、结果或心跳接口。

Action SDK 鉴权方案是本期接口全面保护的前置工作，不能以“内网调用”为理由暂时保留匿名接口。

## 5. 授权模型

### 5.1 主体

系统包含以下授权主体：

- 系统内置账号；
- 普通用户账号；
- 用户权限组；
- Action Component 机器主体。

用户可以加入多个权限组，最终权限为所有有效权限组中有效权限的并集。当前只支持允许权限，不支持显式拒绝。

### 5.2 系统内置账号

系统启动时创建一个固定的系统内置账号和一个系统内置全权限组。

系统内置账号具备以下特性：

- 无视普通 RBAC 和未来的资源隔离，可以访问所有系统资源；
- 只能加入系统内置全权限组；
- 不允许通过业务 API 编辑、禁用、删除、修改密码、修改到期时间或调整权限组；
- 密码仅来源于配置文件；
- 配置文件中的密码变化并重启系统后，系统应按配置重新协调内置账号凭据；
- 配置缺失、使用禁止的默认值或不满足启动安全要求时，非开发环境拒绝启动。

系统内置全权限组具备以下特性：

- 固定拥有 `*`；
- 仅系统内置账号可以使用；
- 不允许分配给普通账号；
- 不允许通过普通权限管理接口删除、禁用、重命名或移除 `*`；
- 启动初始化过程负责检查并恢复其必要状态。

由于系统内置账号作为保底账号存在，本期不判断其他账号是否为最后一个超级管理员。

### 5.3 普通 RBAC

- 普通用户通过一个或多个权限组获得权限；
- 权限组不继承其他权限组；
- 权限组只能引用存在且启用的权限码；
- 普通用户和普通权限组不能拥有 `*`；
- 权限组禁用后立即停止为成员用户提供权限；
- 权限码禁用后立即从所有用户的有效权限中消失；
- 权限校验以服务端计算结果为准，不信任 JWT 或前端缓存中的权限列表。

### 5.4 权限分配范围

每个具备权限管理能力的管理员账号增加独立的“限制权限分配范围”开关。该开关只能由系统内置账号操作。

规则如下：

| 开关状态 | 管理员可分配范围 |
| --- | --- |
| 开启 | 仅限管理员自身当前拥有、有效且允许委派的权限 |
| 关闭 | 可以分配任意已声明且有效的普通权限，即使管理员自身不拥有 |

补充约束：

- 普通管理员永远不能分配 `*`；
- 普通管理员不能把普通用户加入系统内置全权限组；
- 普通管理员不能修改自己的权限分配范围开关；
- 对用户分配权限组和对权限组修改权限时使用同一套范围规则；
- 管理员权限在操作过程中发生变化时，以提交时服务端重新计算的权限为准；
- 开关默认采用安全值“开启”。

### 5.5 未来资源级授权

后续业务资源授权采用：

```text
全局 RBAC 权限 AND（资源所有者 OR 工作区成员关系 OR 系统内置账号）
```

后续需要覆盖的资源至少包括：

- Agent；
- Agent 会话；
- Wiki；
- 行动和行动工作流；
- 采集账号；
- 沙盒；
- 系统后续新增的其他业务资源。

本期只在接口和服务边界预留当前用户上下文，不新增资源所有权字段，不把资源隔离列入本期验收。

## 6. 权限命名规范

### 6.1 页面权限

页面、子页面、独立 Tab 和独立功能组件统一使用：

```text
page:<module>[:<submodule>]:visible
page:<module>[:<submodule>]:access
```

语义：

- `visible`：入口或组件可见；
- `access`：允许进入页面或使用独立功能组件；
- 有 `visible`、无 `access`：展示为禁用状态；
- 无 `visible`：不展示入口；
- 无 `access`：即使通过 URL 或其他方式进入，也必须进入路由级 403 页面；
- 每个已经实现且具有独立功能的 Tab、按钮、图表或组件都必须获得明确权限控制；数据资源操作复用 operation 权限，其他独立 UI 能力使用页面 `visible/access`；
- 尚未实现、仅为视觉占位的组件暂不定义权限码。

示例：

```text
page:system:permissions:visible
page:system:permissions:access
page:system:permissions:users:visible
page:system:permissions:users:access
page:agent:sessions:visible
page:agent:sessions:access
```

### 6.2 操作权限

数据和业务动作统一使用：

```text
operation:<module>:<resource>:read
operation:<module>:<resource>:create
operation:<module>:<resource>:update
operation:<module>:<resource>:delete
operation:<module>:<resource>:execute
```

语义：

- `read`：读取数据；
- `create`：创建资源；
- `update`：修改资源；
- `delete`：删除资源；
- `execute`：触发运行、审批、启停、重置、导入、导出等非普通 CRUD 行为。

批量操作复用对应的单条操作权限，不再建立独立批量权限。

数据列表中的新增、编辑、删除等按钮直接复用数据操作权限，不再为同一个动作重复建立 page 操作权限。独立功能组件可以同时受页面 `visible/access` 和后端数据操作权限约束。

### 6.3 敏感资源的读取拆分

普通内容资源使用统一 `read`。用户、账号和模型配置等敏感资源需要拆分列表读取和详情读取，资源名通过明确的资源标识区分：

```text
operation:system:user-list:read
operation:system:user-detail:read
operation:action:account-list:read
operation:action:account-detail:read
operation:agent:model-config-list:read
operation:agent:model-config-detail:read
operation:agent:model-secret:read
```

列表接口不得因为调用方便而返回只有详情权限才能读取的敏感字段。

### 6.4 用户管理拆分

用户管理至少拆分为以下能力：

```text
operation:system:user-list:read
operation:system:user-detail:read
operation:system:user:create
operation:system:user-profile:update
operation:system:user-group:update
operation:system:user-password:execute
operation:system:user-status:execute
operation:system:user-expiry:update
operation:system:user:delete
```

用户资料编辑权限不得隐式允许重置密码、调整权限组、启停账号或修改到期时间。

### 6.5 业务资源基线

本期为以下资源分别建立权限，至少覆盖适用的 `read/create/update/delete/execute`：

| 模块 | 资源 | 额外要求 |
| --- | --- | --- |
| Agent | Agent | 启动、续聊等运行行为使用 `execute` |
| Agent | 会话 | 读取、终止具体会话分离 |
| Agent | 模型配置 | 列表、详情拆分；模型密钥单独授权 |
| Agent | 技能 | 文件读取、上传、编辑和删除纳入技能权限 |
| Agent | 工作区 | 本期为全局 RBAC，后续接成员关系 |
| Agent | 沙盒 | 创建、销毁、操作分别授权 |
| Action | 行动、蓝图、节点、配置、任务 | 启动行动使用 `execute` |
| Action | 采集账号 | 列表、详情拆分；敏感值读取单独授权 |
| Search | 检索、检索模板 | 搜索执行与模板 CRUD 分开 |
| Target | 重点目标、批注、Wiki | 普通读取与写操作分开 |
| System | 用户、权限组、权限码 | 按管理动作细分 |

实际 YAML 清单应覆盖所有已注册路由，不允许存在没有权限归属的业务端点。

### 6.6 旧权限处理

以下旧设计立即移除：

- 所有 `:view`、`:use` 权限；
- `page:...users:*:use` 等兼容权限；
- 前端 page 权限替代后端 operation 权限的兼容判断；
- 单独的批量新增权限；
- `system:user:disable` 等不符合新格式的权限。

## 7. 权限清单与权限字典

### 7.1 YAML 唯一事实源

仓库中建立一份版本化 YAML 权限清单，作为后端有效标准权限的唯一事实源。清单至少描述：

- 权限码；
- 展示名称；
- 模块和资源分类；
- 描述与标签；
- 是否为系统保留权限；
- 是否可委派；
- 默认启用状态；
- 适用的接口或能力说明。

YAML 用于：

- 生成或加载后端权限注册表；
- 初始化和协调数据库权限字典；
- 校验后端路由引用的权限是否已声明；
- 在 CI 中检查重复、非法命名和未使用权限。

前端不维护另一份手工生成的权威权限清单。前端通过认证接口和权限字典接口获取当前权限及展示元数据，并在路由和组件配置中消费稳定权限码。

### 7.2 标准权限与占位权限

权限字典支持两类权限：

| 类型 | 来源 | 后端授权效力 | 用途 |
| --- | --- | --- | --- |
| 标准权限 | YAML | 有效 | 已被后端接口或服务正式接入的权限 |
| 占位权限 | 前端通过权限字典接口创建 | 无后端放行效力 | 前端页面或组件先行设计时占位 |

占位权限规则：

- 可以进入权限字典并用于前端设计和展示；
- 不得绕过后端有效权限注册表；
- 在后端正式实现对应功能时，必须加入 YAML 并转为标准权限；
- 与现有标准权限同名时不得重复创建；
- 需要在界面中明确标识“占位、尚未接入后端”；
- 占位权限不能是 `*`，也不能伪装成系统保留权限。

### 7.3 启用、禁用、改名和删除

权限字典生命周期规则：

- `enabled=false` 后立即停止产生授权效果；
- 禁用前必须展示受影响的权限组和用户；
- 权限码禁止直接改名；
- 改名必须通过显式迁移操作，提供旧码到新码映射并展示影响范围；
- 删除仍被权限组引用的权限时默认拒绝；
- 如需删除，必须先执行显式迁移或清理操作；
- 删除、禁用和迁移后立即提升相关用户的权限版本；
- 权限字典展示状态不能代替后端有效性判断。

### 7.4 权限组完整性

保存权限组时必须校验：

- 权限码存在；
- 权限码处于启用状态；
- 标准权限或允许分配的占位权限类型合法；
- 操作者拥有所需的权限管理能力；
- 分配范围符合该操作者的独立开关；
- 不包含 `*` 或其他系统保留权限；
- 不对系统内置组执行受保护操作。

数据库应对用户名、权限组标识和权限码等唯一字段建立真正的唯一约束，不能只依赖接口查询避免重复。

## 8. 登录与服务端会话

### 8.1 标准 JWT

访问凭证改为标准 JWT。JWT 至少包含：

- 用户标识；
- 服务端会话标识；
- 签发时间；
- 过期时间；
- 用户授权版本；
- 必要的签发方和用途约束。

权限列表不作为 JWT 中可信的长期授权快照。后端授权以当前用户、权限组、权限字典和服务端会话状态为准。

每个登录会话必须有过期时间，不签发永久不过期 JWT。普通账号可以没有账号到期时间，但登录会话仍按配置的较长有效期过期。

本期不把刷新令牌作为强制验收项；如果不引入刷新令牌，登录会话到期后用户重新登录。后续如引入刷新令牌，必须与同一服务端会话绑定并支持撤销。

### 8.2 服务端会话

系统使用 MongoDB 保存服务端会话状态。每次登录创建独立会话，支持同一账号多端同时登录。登录失败计数、账号临时锁定和按来源速率限制使用 Redis，并使用独立、可配置的 key namespace。

服务端会话至少记录：

- 会话 ID；
- 用户 ID；
- 创建时间、最后活动时间和过期时间；
- 当前状态；
- 登录 IP 和 User-Agent；
- 凭据版本和授权版本；
- 终止时间与终止原因。

每次认证至少验证：

1. JWT 签名、用途和过期时间有效；
2. 服务端会话存在且未终止；
3. 会话与 JWT、用户一致；
4. 用户存在、启用且账号未到期；
5. 凭据版本有效；
6. 授权使用当前有效权限和版本。

### 8.3 会话失效规则

| 事件 | 处理结果 |
| --- | --- |
| 用户主动退出 | 当前会话立即终止 |
| 管理员终止具体会话 | 指定会话立即终止 |
| 用户修改密码或管理员重置密码 | 该用户全部现有会话立即终止 |
| 用户被禁用 | 该用户全部现有会话立即终止 |
| 账号到期 | 该用户全部现有会话立即终止 |
| 用户权限组变化 | 提升授权版本，后续请求立即使用新权限 |
| 权限组权限变化或禁用 | 提升所有受影响用户的授权版本 |
| 权限码禁用、迁移或清理 | 提升所有受影响用户的授权版本 |
| JWT 或服务端会话到期 | 会话失效并要求重新登录 |

后台会话管理需要支持：

- 查看用户的活动会话列表；
- 查看会话的 IP、User-Agent、创建和活动时间；
- 终止某个具体会话；
- 终止某个用户的全部会话。

### 8.4 权限版本刷新

服务端维护用户授权版本。权限来源发生变化时，受影响用户的授权版本立即增加。

前端在以下场景刷新 `/auth/me` 或等价授权上下文：

- 检测到服务端权限版本与本地版本不同；
- 浏览器窗口重新获得焦点；
- API 返回操作级 403；
- 页面首次恢复已有登录状态。

刷新后立即更新菜单、路由、Tab、按钮和其他组件状态。服务端仍必须独立拒绝无权限请求，不能等待前端刷新完成。

### 8.5 临时账号

- 临时账号必须设置账号到期时间；
- 到期时间不得超过配置的最大有效期；
- 临时账号不得设置为永久有效；
- 创建和修改临时账号时均执行相同校验；
- 到期后现有会话立即失效。

### 8.6 登录防护

本期实现：

- 登录失败次数限制；
- 达到阈值后的账号临时锁定；
- 按账号和请求来源进行登录速率限制；
- 锁定阈值、统计窗口、锁定时长和速率阈值均由配置项控制；
- 登录失败响应不泄露用户名是否存在；
- 系统内置账号同样受速率限制，但其解锁和恢复流程必须可运维处理。

本期不实现密码复杂度、密码历史和首次登录强制修改密码。

## 9. 后端授权设计

### 9.1 默认认证

后端路由采用“默认受保护、公开端点显式列出”的策略。新增模块不能依靠开发者记忆逐个补认证。

认证和授权职责分离：

- 认证负责解析 JWT、验证服务端会话和加载当前用户；
- 授权负责计算当前有效权限并校验操作权限；
- 资源授权在后续阶段叠加所有权和成员关系判断；
- Action Component 使用独立机器认证依赖。

### 9.2 授权计算

普通用户的有效权限为：

```text
用户所有已启用权限组中的权限
∩ 当前存在且 enabled=true 的权限字典
∩ 后端标准权限注册表或允许返回前端的占位权限集合
```

其中只有 YAML 标准权限可以作为后端接口放行依据。系统内置账号直接通过普通权限和资源范围校验，但仍必须具有有效登录会话。

### 9.3 接口权限矩阵

每个业务接口必须登记：

- 请求方法和路径；
- 所属模块与资源；
- 所需操作权限；
- 是否属于敏感列表或详情读取；
- 是否还需要执行类权限；
- 未来需要叠加的资源范围条件；
- 调用主体是用户还是 Component。

CI 应检查所有业务路由都已进入权限矩阵。未登记接口不得通过验收。

### 9.4 权限管理保护

权限管理接口除自身 operation 权限外，还必须执行业务规则：

- 防止普通管理员写入 `*`；
- 防止普通管理员操作系统内置账号和系统内置组；
- 校验权限分配范围开关；
- 防止通过用户编辑接口间接执行密码重置、启停或组分配；
- 密码重置、组分配、启停、到期时间修改使用独立接口和权限；
- 禁用和删除权限码前返回影响分析；
- 权限迁移和清理必须显式执行。

## 10. 前端权限设计

### 10.1 权限上下文

前端登录上下文至少包含：

- 当前用户安全展示信息；
- 当前有效权限集合；
- 当前授权版本；
- 当前会话标识和到期信息；
- 系统内置账号标识。

本地缓存仅用于改善体验，不能成为授权事实源。权限工具遇到未知、空值、类型错误或缺失配置时应默认拒绝，并在开发环境给出明确错误。

### 10.2 路由、菜单和组件

- 菜单和 Tab 使用 `visible` 控制显示；
- 路由进入使用 `access` 控制；
- 有 `visible` 无 `access` 时入口显示但禁用；
- 通过 URL 进入无 `access` 页面时进入全局 403；
- 每个 Tab 独立加载数据，不请求无权访问的其他 Tab 接口；
- 独立功能组件按 `visible/access` 控制；
- 数据列表的 CRUD 按钮使用对应 operation 权限；
- 后端未实现的视觉占位组件不提前创建权限。

### 10.3 401 和 403

- 401 或会话失效：清理本地登录状态并跳转登录页；
- 路由级 access 不足：进入全局 403 页面；
- 单个 API 操作返回 403：保留当前页面，在操作位置提示无权执行；
- 操作级 403 同时触发一次权限版本检查和权限刷新；
- 刷新后如当前页面 access 已失效，再由路由守卫进入全局 403；
- 可选数据加载失败不得无条件把整个应用跳转到 403。

### 10.4 敏感配置编辑

账号密码、2FA 密钥和模型 API Key 更新采用“留空表示不修改”：

- 编辑其他字段时不要求前端回传原密钥；
- 留空时后端保留原值；
- 明确填写新值时才更新；
- 不允许使用掩码字符串覆盖真实值；
- 错误信息和前端日志不得包含敏感值。

本期仍保持当前敏感值存储和查看策略，字段加密、Secret Manager 和临时解密查看留作后续。

5. 实现授权版本和凭据版本；

2. 初始化系统内置账号和系统内置组；
3. 实现用户管理权限拆分；
4. 实现权限分配范围开关；
5. 实现 `*` 和系统资源保护；

### 阶段四：业务接口全面接入

3. 执行后端、前端和端到端权限测试；
4. 执行接口权限矩阵覆盖检查；
5. 完成安全回归和验收。

## 15. 需要清理的数据目标

本次不做旧权限兼容或迁移。开发环境执行重构版本前，应备份必要数据后清理以下 MongoDB 集合：

```text
auth_users
auth_groups
auth_permission_codes
```

如果开发过程中已经创建新版服务端会话存储，还需要同步清理：

- 新增的登录会话集合；或
- Redis 中属于新版登录会话、登录失败计数、锁定和速率限制的专用 key namespace。

无需清理 Agent、Wiki、Action、Search、Annotation 等业务集合。本期尚未实现资源所有权，因此清理用户数据不会执行资源归属迁移。

清理后由启动初始化流程重新创建：

- 系统内置账号；
- 系统内置全权限组；
- YAML 中声明的标准权限字典；
- 系统内置账号与系统内置组的唯一绑定关系。

清理脚本是独立运维脚本，不得导入到 API 路由、应用 lifespan、前端或普通业务服务中。脚本必须仅允许在显式 `ENVIRONMENT=development` 时运行，在执行前显示目标数据库、集合和 Redis namespace，并要求操作者输入显式确认，不得隐式作用于生产数据库。本轮只交付并测试脚本，不自动对当前配置的 `csi_db` 执行清理。

## 18. 风险与约束

- Action Component 已确定采用“一次性 Redis 引导码交换独立短期 JWT”的形式；剩余风险是组件 JWT 到期时间需要结合最长任务运行时间持续校准；
- 本期不实现资源隔离，因此拥有某项全局业务权限的用户仍可能访问该类型下的全部资源；
- 本期继续明文保存并按现有方式查看敏感配置，相关风险不因权限重构而消失；
- 系统内置账号密码由配置和重启管理，配置文件本身必须受到严格的文件权限和交付保护；
- 权限码直接清库重建会使所有开发账号和权限组失效，执行前需确认目标数据库；
- YAML 标准权限和数据库占位权限并存时，管理界面必须清楚展示二者效力差异。
